Full-Text Search
Postgres-native tsvector search across part names, descriptions, and numbers. GIN-indexed for sub-millisecond lookups on 10K+ parts.
CREATE INDEX ... USING GIN (to_tsvector(...))
🌳
BOM Tree Traversal
Recursive CTEs with cycle detection traverse multi-level assemblies. get_bom_tree() and where_used() handle 10-level deep BOMs in <50ms.
WITH RECURSIVE ... NOT (id = ANY(path))

AI Agent (17 Tools)
LangGraph ReAct agent with org-scoped context injection. Creates parts, traverses BOMs, manages ECOs, generates impact visualizations.
Parts(5) + BOM(5) + ECO(7) + Viz(2)
🔒
Multi-Tenant RLS
Row-Level Security on every table. Org isolation enforced at database level. Even the AI agent can't cross tenant boundaries.
80+ policies · get_user_org_ids()
⚙️
Workflow State Machines
Database triggers enforce valid status transitions for Parts and ECOs. Multi-step approval with FOR UPDATE locking prevents race conditions.
validate_part_transition() + process_review_decision()
📊
Flexible Attributes
JSONB attributes per part with GIN indexing. Category-specific schemas defined in attribute_schemas table. No rigid column structure.
attributes jsonb + jsonb_path_ops index
✅
Why this works for Phase 1: PostgreSQL's recursive CTEs ARE graph traversals on relational storage. A typical BOM (5-8 levels, thousands of parts) runs in <50ms. You get ACID transactions, battle-tested security, and the entire Supabase ecosystem — for $25/mo.

🧠
pgvector Extension
Add a vector(1536) column to parts table. Supabase supports pgvector natively — no separate database needed. HNSW index for fast approximate nearest-neighbor search.
ALTER TABLE parts ADD COLUMN embedding vector(1536)
🔄
Auto-Embedding Pipeline
Postgres trigger on INSERT/UPDATE calls an Edge Function that embeds name + description + attributes into a vector. Zero manual work — embeddings stay in sync.
AFTER INSERT OR UPDATE ON parts → embed()
🔎
Hybrid Search
Combine keyword FTS + vector similarity for best results. Exact part number matches rank first, then semantic similarity fills the gaps. Agent gets a new semantic_search tool.
RRF(tsvector_rank, 1 - cosine_distance)
📄
Document Management
Supabase Storage for file uploads. Documents linked to parts and ECOs. Revision tracking with immutable storage. Agent gets 4 new document tools.
bucket: documents · path: {org_id}/{doc_id}/
⚡
Why pgvector, not a separate vector DB: It's a Postgres extension running in the SAME database. No sync layer, no extra infra, no latency. Your RLS policies still apply. Enable it with one SQL command: CREATE EXTENSION vector;. Supabase includes it on all plans.
💰
Cost impact: $0 additional infrastructure. Embedding API calls (OpenAI text-embedding-3-small) cost ~$0.02 per 1M tokens. A 10K-part catalog embeds for under $1.

Phase 1: Multi-Tool Agent Call
-- "What's affected if Supplier X has issues?" -- Agent needs 4 separate tool calls: 1. search_parts(supplier="X") 2. FOR EACH part: get_where_used(part_id) 3. get_bom_tree(each assembly) 4. list_recent_ecos(affected parts) -- 4 round-trips, manual correlation
Phase 3: Single Graph Traversal
-- Same question, one query: SELECT * FROM ag_catalog.cypher( 'plm_graph', $$ MATCH (s:Supplier {name:'X'}) -[:SUPPLIES]->(p:Part) <-[:CONTAINS*]-(assy:Part) <-[:AFFECTS]-(eco:ECO) RETURN p, assy, eco $$ ) AS (p agtype, assy agtype, eco agtype); -- 1 query, full context, <100ms
🕸️
Apache AGE Extension
OpenCypher graph queries running INSIDE PostgreSQL. No separate Neo4j server. Same ACID transactions, same RLS security. Graph and relational data in one DB.
CREATE EXTENSION age; · Cypher via SQL
🗺️
Materialized Impact Graph
Pre-computed BOM traversals refreshed on changes. Agent reads the cache instead of running recursive CTEs live. Sub-millisecond impact analysis.
REFRESH MATERIALIZED VIEW CONCURRENTLY
🔗
Cross-Entity Reasoning
One query traverses: Part → BOM → ECO → Supplier → Regulation. The AI agent answers complex "what-if" questions in a single tool call instead of 4-5.
MATCH path = (a)-[*1..6]->(b) RETURN path
🏭
Supplier & Compliance Graph
New entities: Suppliers, Regulations, Certifications modeled as graph nodes. Enables "which products are non-compliant if regulation Z changes?" queries.
(:Part)-[:GOVERNED_BY]->(:Regulation)
🎯
Why Apache AGE over Neo4j: AGE runs as a Postgres extension. No separate server, no sync layer, no second auth system. Your RLS policies, ACID transactions, and existing queries all continue to work. You add graph capabilities without abandoning your proven foundation.
💡
When to upgrade to standalone Neo4j: Only if your graph exceeds 10M+ edges with real-time 10+ hop traversals. Most PLM systems never reach this. If they do, you migrate the graph layer only — core PLM stays on Postgres.
